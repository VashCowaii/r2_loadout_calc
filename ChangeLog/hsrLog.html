This is extremely handy for maintaining specific rotations, while also being flexible to allow you as the user to specify conditions that will react to the state of the battle instead of needing to manually define your rotations in every possible scenario bc fuck that.
    • A drawback to how we have done things is that while the standard conditions work as a glorified on/off switch for you, you couldn't ever actually dictate WHO got selected with certain abilities.
    • A scenario for why this sucks would be as follows:
      • Say you have Tingyun and Robin, and Robin is ALMOST ready to ult, but you have 5AV left before the battle ends, in the prior settings of the calculator even if Tingyun's ult was ready to go she would always try to target char1, meaning Robin wouldn't get enough energy, and would fail to ult before the battle ended.
    • NOW, however, there is a new tab under Rotations called Targeting, and in here you can how define a target priority list for each ability that warrants it.
      • Please note that target logic is a limited release, and for now ONLY the characters listed below will have access to it on abilities that target allies.
        • Sparkle
        • Bronya
        • Trailblazer - Remembrance
        • DHPT
        • Sunday
      • Abilities that are AOE targeting obv don't need target logic
      • Enemy targeting for damaging abilities is in the works but that is a little more complicated on the backend of things so that's not ready yet.
    • Target listings will each consist of 2 parts:
      • The Filter: This lets you filter the target pool available to that ability, and if no target can match your filters, that condition will fail, but if you DO find targets to return from the filters then you move on to the second part
      • The Check: This is where if you found targets, you can set conditions for their use. So say you want Bronya to target character 1 UNLESS she IS character 1, then you can set conditions for that and if she were character 1 then target listing would fail and move down the list.
    • Target listings will always be evaluated from top to bottom, and every ability that HAS target logic will also have a fallback target in case you set some really fucking stupid conditions.
      • In most cases for ally targeting, the fallback target will be character 1.

    • Added roll restrictions to each substat relic. This is a little weird so let me explain.
      • As like before, when a stat is input, it will estimate on average the roll count that it would take to reach the value that you input on that substat.
        • As an example, a SPD value of exactly 13 would estimate 5 rolls to reach that point because with mixed high/mid/low values that is realistic
      • However, estimates don't really let us enforce a 5 maximum roll limit, so what would happen if someone enters in a higher value on the next substat?
        • In this case, it would look at the first value again, and it knows that on SPD the highest value per roll possible is 2.6, and 2.6 * 4 + (base roll 2.6) = 13 meaning that 4 rolls is the lowest possible roll value that could contain 13. At this point, the roll value state will change from the estimated 5 to the bounded 4
        • Now the next value that you input will be evaluated in the same way. If you happened to input like 15% crit rate, it will be restricted instead to the highest possible crit rate value using 1 roll since only 1 roll was leftover from the SPD substat on this relic
        • From top to bottom, substats will prioritize roll counts and restrict any subsequent rolls based on how many are leftover from those upper substats
      • In this way, we can ensure that users are not setting 4 rolls of crit rate at 15% each.
